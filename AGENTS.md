# Codex 项目规则

## 页面刷新保留售前采集参数

5174 主方案页刷新后，必须保留用户上一次填写的售前采集参数和手动数量调整，不得回到默认普通教室参数。

实现要求：

- 使用本地浏览器存储保存售前采集草稿。
- 保存内容至少包括：使用场景、需求、房间尺寸、现场条件、已有设备、声学环境、中央空调 / 利旧音箱点位、手动数量调整。
- 页面初始化时优先读取草稿；草稿损坏或不存在时才使用默认初始参数。
- 导入 JSON 或用户修改参数后，要继续自动更新草稿。

## 乱码、Bug 和代码问题处理边界

工作过程中发现以下问题时，必须先记录到 `logs/execution_log.md` 和 `logs/retrospective.md`：

- 乱码、残留文案、错误文案
- 页面可见文案、图纸标注、复勘提醒、按钮 / 表格字段中出现损坏字符或疑似乱码
- 明显 bug、非阻塞代码问题、潜在 UI 问题
- 无用代码、临时调试代码、旧逻辑残留
- 类型检查、构建、页面运行中暴露但不阻塞当前校准的问题

记录后不要在规则校准过程中顺手清理或顺手修复，避免把业务规则带偏。

如果损坏字符出现在“避让、复勘提醒、安装建议、风险提示”等规则相关文案中，默认只记录问题和影响位置，不在当前规则校准中顺手改业务逻辑；每天收工时统一按日志清理文案，并确认没有改变音箱 / 阵麦规则。

只有当问题会导致当前功能无法运行、类型检查无法通过、页面无法打开、或用户明确要求立即修复时，才可以当场处理；处理后也要写入日志，说明为什么没有等到收工统一处理。

每天工作结束时，按“每日收工流程”的第三步统一处理这些日志里记录的问题。

## 上下文压缩后的第一步

每当发生上下文压缩、线程恢复、隔天继续、接手他人总结、或明显缺少前文时，Codex 必须先读取：

- `AGENTS.md`
- `logs/execution_log.md`
- `logs/retrospective.md`

读完底层规则和日志后，先恢复当前项目的工作边界，再继续分析、改代码或回答问题。

发生上下文压缩前，如果当前线程里已经做过校准规则确认、规则改动、边界约定、踩坑修正或重要验证，必须先把这些工作写入 `logs/execution_log.md` 和 `logs/retrospective.md`，再进入压缩后的继续工作。若压缩是系统自动发生、无法提前写日志，恢复后的第一步必须补写压缩前遗漏的校准规则和工作边界记录。

读取 `AGENTS.md` 或包含中文的项目规则文件时，优先使用 Node / UTF-8 方式读取；如果 PowerShell `Get-Content` 显示乱码，但 Node UTF-8 读取正常，归类为终端显示编码问题，不要误判为文件损坏，也不要因此重写整份规则文件。

## 规则改动确认边界

涉及音箱选型、音箱点位、音箱数量、阵麦点位、阵麦数量的规则改动，必须先说明：

- 当前触发了什么规则
- 为什么当前结果不对或不直观
- 准备改哪条规则
- 改完会影响哪些场景

只有用户明确确认后，才能修改这些规则。不要把用户的追问、质疑、或“删掉”自动理解成可以直接改代码。

## 桌面端 / 移动端样式隔离

5174 网页端是主工作页面，5177-5179 是移动预览入口，5180 是音曼独立预览入口。更改移动端样式、移动端标题、移动端布局或移动端预览壳时，不得影响 5174 网页端的桌面布局，也不得把 5180 音曼桌面预览误套成移动端。

实现要求：

- 移动预览专属样式必须优先写在 `.mobilePreviewMode` 作用域内，且只能由移动预览入口触发。
- 真正手机窄屏样式必须放在明确的窄屏媒体查询里，例如 `@media (max-width: 560px)`。
- 不得把移动端字体缩放、单列布局、窄屏按钮布局、移动预览宽度限制写成全局规则。
- 修改移动端后，必须同时检查 5174 桌面宽度和 5177 移动预览，确认桌面端没有被移动端规则带偏。

## 终端默认规则

以后在本项目里执行命令时，默认优先使用 PowerShell 7（`pwsh`）。

- 需要手动指定 shell 或启动新终端时，优先用 `pwsh`，不要默认用 Windows PowerShell 5.1。
- 如果当前环境没有 `pwsh` 或 PowerShell 7 无法启动，才回退到系统默认 PowerShell，并在回复或日志中说明。
- 涉及中文文件、日志、AGENTS.md、报告文案读取时，仍优先按 UTF-8 读取，避免把终端编码显示问题误判为文件损坏。

## Git 自动存档与回滚规则

本项目已经初始化为 Git 仓库，并推送到 GitHub：`https://github.com/zhanghao1556/shouqianAPP.git`。

以后 Codex 在本项目中完成代码、脚本、规则、日志、文档等有效改动后，默认自动执行 Git 存档：

1. 先运行 `git status -sb`，确认工作区改动范围。
2. 如果改动都属于当前用户请求，自动提交并推送到 GitHub；可使用 `scripts/git-checkpoint.ps1 -Message "说明"` 或等效的 `git add -A && git commit && git push`。
3. 提交信息要简短说明本次改动，例如 `document git checkpoint automation`、`fix mobile header title`。
4. 提交前不得把 `.codex-backups`、`node_modules`、`dist`、`outputs`、`output`、`work`、`docx_2` 等本地大目录加入 Git；这些目录应继续由 `.gitignore` 排除。
5. 如果工作区存在与当前任务无关、用户可能手动改过的文件，不得自动 `git add -A`；必须先说明差异并询问哪些文件要进入提交。
6. 每次成功提交 / 推送后，在回复中告知提交号和远端分支状态。

每天工作结束或执行打包 / 发布流程时，必须额外保存一个带时间的 Git 历史存档点：

- 每日收工完成后运行 `scripts/git-checkpoint.ps1 -Kind daily`。
- 打包 / 发布包生成并验证完成后运行 `scripts/git-checkpoint.ps1 -Kind release`。
- 这两类存档点的提交信息必须包含日期和分钟，例如 `daily checkpoint 2026-07-08 20:58`、`release checkpoint 2026-07-08 21:03`。
- 即使当时没有新的可提交文件改动，也要允许创建空提交，确保 Git 历史里能看到当天或本次发布的明确存档点。

回滚属于高风险操作：

- `git restore .`、`git reset --hard`、`git clean`、`git push --force` 等会丢弃或改写历史的命令，必须先运行 `git status -sb` 和 `git log --oneline -5`，说明会影响哪些内容，并取得用户明确确认后才能执行。
- 用户只说“回滚”时，默认先分析当前改动和可回滚点，不直接执行破坏性命令。
- 回滚前如果存在未提交改动，优先建议先做一个临时提交或备份分支，避免误删用户工作。

## 每日收工流程

每天结束工作时，Codex 必须自动执行四步：

1. 先写日志：更新 `logs/execution_log.md` 和 `logs/retrospective.md`，记录当天做了什么、确认了什么规则、犯了什么错误、发现了哪些乱码 / bug / 残留问题、下一步从哪里继续。
2. 再做备份：保存一个新的当前状态快照到 `.codex-backups`，确认新快照成功后，删除之前的旧快照，只保留最新一个可回滚快照。
3. 最后清理代码：根据日志中记录的问题做安全代码清理和检查，包括类型检查、无用代码 / 残留文案扫描、明显临时调试残留清理。不得借“清理代码”擅自修改音箱选型、音箱点位、音箱数量、阵麦点位、阵麦数量等业务规则。
4. 收工 Git 存档：运行 `scripts/git-checkpoint.ps1 -Kind daily`，在 Git 历史中保存一个带日期和分钟的每日存档点。

## 发布版本流程

当用户要求“打包”“发布新版本”“发布包”“发版”时，默认这就是当天工作结束的最后一步。

发布前必须先执行每日收工流程：

1. 先读取 `logs/execution_log.md` 和 `logs/retrospective.md`，确认当天已完成事项、待清理问题、已确认规则和风险边界。
2. 按“每日收工流程”写日志、做备份、清理代码和运行必要检查。
3. 确认日志、备份、清理和检查完成后，最后才生成发布包。
4. 发布包生成并验证完成后，运行 `scripts/git-checkpoint.ps1 -Kind release`，在 Git 历史中保存一个带日期和分钟的发布存档点。

不得跳过每日收工流程直接打包。发布包必须放在整个流程最后生成。

所有发布版本都必须同步到 GitHub：

- 发布包目录和发布 zip 属于最终交付物，必须随发布 checkpoint 一起提交并推送到 `zhanghao1556/shouqianAPP`。
- 同一天多次打包发布时，不得覆盖旧发布包；发布目录和 zip 名称必须在日期后追加递增序号，例如 `260709-1`、`260709-2`、`260709-3`。
- 发布验证脚本必须针对最新发布目录运行，不能写死旧日期。
- 如果 GitHub 推送失败，必须明确告诉用户本地提交号、ahead 数量和失败原因；网络恢复后要继续推送，直到 `main` 与 `origin/main` 同步。

以后用户要求“打包发布”“发布新版本”“发布包”“发版”且未特别限定品牌时，默认同时发布音翼和音曼两个版本：

- 音翼发布包使用音翼品牌、绿色 / 白色基调、音翼阵麦与既有音翼资源。
- 音曼发布包使用音曼品牌、蓝色 / 白色基调、音曼 logo、音曼阵麦点位图与拓扑图资源。
- 两个品牌发布包都要按同一天多次打包的递增编号规则生成，不得互相覆盖。
- 两个品牌发布包都要完成对应验证并随发布 checkpoint 一起提交、推送到 GitHub。
- 若用户明确指定只发布某一个品牌，才按用户指定品牌单独发布。

## Backup Retention

Daily closing backups must keep only the newest one valid `.codex-backups` snapshot zip file. Create and verify the new snapshot first, then delete only backup zip files older than the newest valid snapshot.

## 发布包售前采集清空规则

发布版本、内部测试版、客户可打开的单文件 HTML 不得读取开发页或上一轮测试留下的售前采集草稿。

- 5174 开发页仍按“页面刷新保留售前采集参数”执行，方便当天连续校准。
- 发布包首次打开必须使用干净采集表：项目、客户、房间尺寸、现场条件、已有设备、中央空调 / 利旧点位、备注、手动数量调整都不得继承本机草稿。
- 发布脚本必须显式标记发布态，应用根据发布态跳过开发草稿读取和写入。

## 发布版客户可见文案规则

发布版所有客户可见文案只写“什么参数会影响什么结果”，不要写为什么、推导过程、算法解释或内部规则原因。

- 复勘提醒只保留影响关系，例如“混响会影响阵麦拾音清晰度”。
- 不在发布版复勘提醒里写“因为、由于、按规则、当前算法、必要时”等解释型内容。
- 内部校准台、日志和代码注释可以保留原因；客户可见输出不保留原因。

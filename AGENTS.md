# Codex 项目规则

## 页面刷新保留售前采集参数

5174 主方案页刷新后，必须保留用户上一次填写的售前采集参数和手动数量调整，不得回到默认普通教室参数。

实现要求：

- 使用本地浏览器存储保存售前采集草稿。
- 保存内容至少包括：使用场景、需求、房间尺寸、现场条件、已有设备、声学环境、中央空调 / 利旧音箱点位、手动数量调整。
- 页面初始化时优先读取草稿；草稿损坏或不存在时才使用默认初始参数。
- 导入 JSON 或用户修改参数后，要继续自动更新草稿。

## 乱码、Bug 和代码问题处理边界

工作过程中发现以下问题时，必须先记录到 `logs/execution_log.md` 和 `logs/retrospective.md`：

- 乱码、残留文案、错误文案
- 页面可见文案、图纸标注、复勘提醒、按钮 / 表格字段中出现损坏字符或疑似乱码
- 明显 bug、非阻塞代码问题、潜在 UI 问题
- 无用代码、临时调试代码、旧逻辑残留
- 类型检查、构建、页面运行中暴露但不阻塞当前校准的问题

记录后不要在规则校准过程中顺手清理或顺手修复，避免把业务规则带偏。

如果损坏字符出现在“避让、复勘提醒、安装建议、风险提示”等规则相关文案中，默认只记录问题和影响位置，不在当前规则校准中顺手改业务逻辑；每天收工时统一按日志清理文案，并确认没有改变音箱 / 阵麦规则。

只有当问题会导致当前功能无法运行、类型检查无法通过、页面无法打开、或用户明确要求立即修复时，才可以当场处理；处理后也要写入日志，说明为什么没有等到收工统一处理。

每天工作结束时，按“每日收工流程”的第三步统一处理这些日志里记录的问题。

## 上下文压缩后的第一步

每当发生上下文压缩、线程恢复、隔天继续、接手他人总结、或明显缺少前文时，Codex 必须先读取：

- `AGENTS.md`
- `logs/execution_log.md`
- `logs/retrospective.md`

读完底层规则和日志后，先恢复当前项目的工作边界，再继续分析、改代码或回答问题。

发生上下文压缩前，如果当前线程里已经做过校准规则确认、规则改动、边界约定、踩坑修正或重要验证，必须先把这些工作写入 `logs/execution_log.md` 和 `logs/retrospective.md`，再进入压缩后的继续工作。若压缩是系统自动发生、无法提前写日志，恢复后的第一步必须补写压缩前遗漏的校准规则和工作边界记录。

读取 `AGENTS.md` 或包含中文的项目规则文件时，优先使用 Node / UTF-8 方式读取；如果 PowerShell `Get-Content` 显示乱码，但 Node UTF-8 读取正常，归类为终端显示编码问题，不要误判为文件损坏，也不要因此重写整份规则文件。

## 规则改动确认边界

涉及音箱选型、音箱点位、音箱数量、阵麦点位、阵麦数量的规则改动，必须先说明：

- 当前触发了什么规则
- 为什么当前结果不对或不直观
- 准备改哪条规则
- 改完会影响哪些场景

只有用户明确确认后，才能修改这些规则。不要把用户的追问、质疑、或“删掉”自动理解成可以直接改代码。

## 实现与验收节奏

用户已经明确确认功能或规则方案后，优先尽快完成一个可运行版本，不要在交付前长时间代替用户逐项操作页面验收。

- 普通开发任务完成后，Codex 先运行类型检查、目标规则测试和生产构建等自动化基础自检。
- 自动化基础自检通过后即可把页面交给用户做业务体验和视觉验收；用户反馈问题后再定点修正。
- 除非用户明确要求 Codex 代验、问题只能通过浏览器复现、或进入正式打包发布流程，否则不做耗时的全页面逐项点击、重复截图和多轮人工浏览器验收。
- 此节奏不降低发布验证要求；用户要求打包发布时，仍必须执行“发布版本流程”规定的双品牌隔离、业务输出一致性和移动端验证。

## 规则校准通用化

用户指出的具体项目、房间尺寸或当前页面结果只作为复现问题的样例，不得围绕单个场景增加补丁式规则。校准时必须追溯产生该结果的共用底层规则，并让同类型问题在所有适用场景中一起得到改善。

- 不得把客户名称、单个项目尺寸、某一张图的固定坐标或当前案例专属条件写进正式算法。
- 优先校准覆盖目标、分区责任、距离、角度、避让、容量、层级和连接关系等通用工程规则，而不是新增只命中当前场景的分支。
- 只有不同场景确实存在声学原理、产品能力或安装方式差异时，才允许保留场景分层；提出方案时必须说明差异依据、共用规则和影响范围。
- 推荐规则方案时必须说明它会解决哪些同类型问题、影响哪些既有场景，并用当前案例生成预览图；实现后还要回归代表性边界场景，防止只把当前图片调正确。

## 规则改动图纸预览

以后 Codex 推荐修改点位、拓扑、接线、报告绘图等业务规则时，必须在请求用户确认方案之前，先按当前问题场景生成一张拟调整后的对应图纸，让用户直接看图校准。

- 修改点位图规则时生成拟调整点位图；修改拓扑规则时生成拟调整拓扑图；修改接线规则时生成拟调整接线图；修改报告绘图或取景规则时生成对应报告页预览。
- 一次方案会影响多张图时，必须把所有受影响图纸都生成预览，不得只展示其中一张。
- 预览必须明确标记“拟调整预览 / 尚未写入正式规则”，并使用当前问题场景的真实房间参数、设备数量和既有点位作为基线。
- 生成预览不得提前修改正式业务规则、当前开发页结果或发布包；用户明确确认后，才能实现规则并重新生成正式图纸验证。
- 如果原渲染链路暂时无法直接生成预览，应使用不改变正式规则的本地模拟图展示同一组点位、角度、连接和覆盖关系；仍无法形成可读图片时，必须说明原因，不能只用文字要求用户确认。

## 图纸画布与避让规则

点位图、拓扑图、接口接线图和报告图纸的可见画布不得超过当前浏览器或所属内容容器的可用宽度，不得用超宽 SVG、固定最小宽度或横向滚动作为布局兜底。

- 图纸宽度以浏览器内容区为硬上限；设备或标注增多时优先增加纵向高度，不得把页面横向撑宽。
- 一级设备保持中心；二级设备在可用宽度内均匀围绕一级设备；三级设备优先在所属二级设备周围的空白位置安置。
- 节点、线材标签和接口文字必须参与碰撞检查。可通过上下、左右空白通道避让；不得压住设备、接口文字或其他标签。
- 宽度不足以维持环绕布局时，改为上下分组或纵向分层，保持层级和父子关系；不得把文字缩小到不可读。
- 响应式回归至少检查画布宽度等于容器宽度、无节点越界、无节点重叠、无页面横向溢出，以及参数变化后重新排布仍满足上述条件。

## 音曼大圆盘阵麦处理器硬规则

音曼大圆盘阵麦只能连接内部型号 `AJ350`，客户侧统一显示“高性能处理器”。

- 1 只大圆盘阵麦使用 AJ350 `A1`；2 只分别使用 `A1 / A2`，仍只配置 1 台 AJ350。
- 浏览器草稿或历史方案中残留的双麦 / 六麦处理器手动档位对大圆盘阵麦无效，设备选择、接口接线和拓扑必须强制回到 AJ350。
- 大圆盘阵麦方案的设备清单只显示“高性能处理器”，不得显示双麦处理器或六麦处理器候选行，也不得提供处理器切换入口。
- 超过 2 只大圆盘阵麦按阵麦能力上限产生硬风险，不得通过改成双麦或六麦处理器绕过产品限制。

## 桌面端 / 移动端样式隔离

5174 网页端是主工作页面，5177-5179 是移动预览入口，5180 是音曼独立预览入口。更改移动端样式、移动端标题、移动端布局或移动端预览壳时，不得影响 5174 网页端的桌面布局，也不得把 5180 音曼桌面预览误套成移动端。

实现要求：

- 移动预览专属样式必须优先写在 `.mobilePreviewMode` 作用域内，且只能由移动预览入口触发。
- 真正手机窄屏样式必须放在明确的窄屏媒体查询里，例如 `@media (max-width: 560px)`。
- 不得把移动端字体缩放、单列布局、窄屏按钮布局、移动预览宽度限制写成全局规则。
- 修改移动端后，必须同时检查 5174 桌面宽度和 5177 移动预览，确认桌面端没有被移动端规则带偏。

## 终端默认规则

以后在本项目里执行命令时，默认优先使用 PowerShell 7（`pwsh`）。

- 需要手动指定 shell 或启动新终端时，优先用 `pwsh`，不要默认用 Windows PowerShell 5.1。
- 在本机不得裸调用户目录下的 WindowsApps `pwsh` alias；优先使用真实 PowerShell 7 路径。传统安装路径是 `C:\Program Files\PowerShell\7\pwsh.exe`，当前 Store 版路径是 `C:\Program Files\WindowsApps\Microsoft.PowerShell_7.6.3.0_x64__8wekyb3d8bbwe\pwsh.exe`。
- 如果当前环境没有 `pwsh` 或 PowerShell 7 无法启动，才回退到系统默认 PowerShell，并在回复或日志中说明。
- Codex 托管执行器若在启动命令前把显式 `pwsh` / `powershell` shell 强制解析为用户目录下无权限的 WindowsApps alias，并报 `CreateProcessAsUserW failed: 5`，不要继续重试 `pwsh`、传统安装路径或 Store 路径；本轮显式指定 `cmd.exe` 作为兼容 shell，中文 / JSON / 复杂文本处理改用 Node UTF-8 脚本，并在日志中记录一次回退原因。该情形属于执行器启动限制，不代表项目文件或 PowerShell 7 安装损坏。
- 涉及中文文件、日志、AGENTS.md、报告文案读取时，仍优先按 UTF-8 读取，避免把终端编码显示问题误判为文件损坏。

## Git 自动存档与回滚规则

本项目已经初始化为 Git 仓库，并推送到 GitHub：`https://github.com/zhanghao1556/shouqianAPP.git`。

以后 Codex 在本项目中完成代码、脚本、规则、日志、文档等有效改动后，默认自动执行本地 Git 存档，但不得默认推送到 GitHub：

1. 先运行 `git status -sb`，确认工作区改动范围。
2. 如果改动都属于当前用户请求，自动本地提交保存；可使用 `scripts/git-checkpoint.ps1 -Message "说明"` 或等效的 `git add -A && git commit`。
3. 提交信息要简短说明本次改动，例如 `document git checkpoint automation`、`fix mobile header title`。
4. 提交前不得把 `.codex-backups`、`node_modules`、`dist`、`outputs`、`output`、`work`、`docx_2` 等本地大目录加入 Git；这些目录应继续由 `.gitignore` 排除。
5. 如果工作区存在与当前任务无关、用户可能手动改过的文件，不得自动 `git add -A`；必须先说明差异并询问哪些文件要进入提交。
6. 每次成功提交后，在回复中告知提交号、本地 ahead 数量和远端分支状态。
7. 默认不执行 `git push`。GitHub 同步由用户手动点击桌面 `上传到GitHub.cmd` 或明确要求 Codex 推送后再执行。

每天工作结束或执行打包 / 发布流程时，必须额外保存一个带时间的 Git 历史存档点：

- 每日收工完成后运行 `scripts/git-checkpoint.ps1 -Kind daily`。
- 打包 / 发布包生成并验证完成后运行 `scripts/git-checkpoint.ps1 -Kind release`。
- 这两类存档点的提交信息必须包含日期和分钟，例如 `daily checkpoint 2026-07-08 20:58`、`release checkpoint 2026-07-08 21:03`。
- 即使当时没有新的可提交文件改动，也要允许创建空提交，确保 Git 历史里能看到当天或本次发布的明确存档点。

回滚属于高风险操作：

- `git restore .`、`git reset --hard`、`git clean`、`git push --force` 等会丢弃或改写历史的命令，必须先运行 `git status -sb` 和 `git log --oneline -5`，说明会影响哪些内容，并取得用户明确确认后才能执行。
- 用户只说“回滚”时，默认先分析当前改动和可回滚点，不直接执行破坏性命令。
- 回滚前如果存在未提交改动，优先建议先做一个临时提交或备份分支，避免误删用户工作。

## 每日收工流程

每天结束工作时，Codex 必须自动执行四步：

1. 先写日志：更新 `logs/execution_log.md` 和 `logs/retrospective.md`，记录当天做了什么、确认了什么规则、犯了什么错误、发现了哪些乱码 / bug / 残留问题、下一步从哪里继续。
2. 再做备份：保存一个新的当前状态快照到 `.codex-backups`，确认新快照成功后，删除之前的旧快照，只保留最新一个可回滚快照。
3. 最后清理代码：根据日志中记录的问题做安全代码清理和检查，包括类型检查、无用代码 / 残留文案扫描、明显临时调试残留清理。不得借“清理代码”擅自修改音箱选型、音箱点位、音箱数量、阵麦点位、阵麦数量等业务规则。
4. 收工 Git 存档：运行 `scripts/git-checkpoint.ps1 -Kind daily`，在 Git 历史中保存一个带日期和分钟的每日存档点。

## 发布版本流程

当用户要求“打包”“发布新版本”“发布包”“发版”时，默认这就是当天工作结束的最后一步。

发布前必须先执行每日收工流程：

1. 先读取 `logs/execution_log.md` 和 `logs/retrospective.md`，确认当天已完成事项、待清理问题、已确认规则和风险边界。
2. 按“每日收工流程”写日志、做备份、清理代码和运行必要检查。
3. 确认日志、备份、清理和检查完成后，最后才生成发布包。
4. 发布包生成并验证完成后，运行 `scripts/git-checkpoint.ps1 -Kind release`，在 Git 历史中保存一个带日期和分钟的发布存档点。

不得跳过每日收工流程直接打包。发布包必须放在整个流程最后生成。

所有发布版本都必须本地保存，GitHub 同步由用户手动脚本完成：

- 发布包目录和发布 zip 属于最终交付物，必须随发布 checkpoint 一起本地提交保存。
- 同一天多次打包发布时，不得覆盖旧发布包；发布目录和 zip 名称必须在日期后追加递增序号，例如 `260709-1`、`260709-2`、`260709-3`。
- 发布验证脚本必须针对最新发布目录运行，不能写死旧日期。
- 发布完成后默认不推送 GitHub；必须明确告诉用户本地提交号和 ahead 数量，并提醒用户网络正常时点击桌面 `上传到GitHub.cmd` 手动同步。

## 发布版本号与交付文档

- `package.json` 的 `version` 是正式发布版本的唯一来源。页面构建常量、PDF 报告载荷、单文件 HTML 标记、发布目录、zip、README、产品说明书和验证脚本必须自动读取同一版本，不得在各脚本中重复写死版本号。
- 正式执行 `release:yinyi`、`release:yinman` 或 `release:all` 时，先运行版本准备脚本。当前版本已完成目标品牌发布时，默认自动递增次版本，例如 `2.0.0 -> 2.1.0`；当前版本尚有任一目标品牌未发布时复用当前版本，确保失败重试不会重复跳号。
- 用户明确要求大版本或补丁版本时，通过版本准备脚本的 `major` / `patch` 选项执行；一次双品牌发布只能更新一次版本，音翼和音曼必须使用相同版本。
- 普通构建、验证、浏览器复核和发布失败后的内部重跑不得自行更新版本。版本更新必须同时写入 `package.json` 和 `package-lock.json`，并随本次发布 checkpoint 保存。
- 新发布包必须恰好包含 3 个客户交付文件：品牌单文件 HTML、`README-打开说明.txt`、`品牌AI售前工具-版本-产品说明书.md`。不得再生成、复制或打包“软件大纲”“软件开发大纲”及内部版本路线。
- 产品说明书只写客户可使用的产品用途、运行环境、基本操作、输出内容、注意事项和技术支持，不写内部算法原因、规则推导、校准过程、未来开发路线或具体产品型号。
- 发布验证必须同时检查目录文件清单与产品说明书内容：版本一致、品牌隔离、通用名称合规、没有软件大纲、没有内部测试文案，并确认 zip 仅包含上述 3 个文件。

## 发布版客户可见文案规则

发布版所有客户可见文案只写“什么参数会影响什么结果”，不要写为什么、推导过程、算法解释或内部规则原因。

- 复勘提醒只保留影响关系，例如“混响会影响阵麦拾音清晰度”。
- 不在发布版复勘提醒里写“因为、由于、按规则、当前算法、必要时”等解释型内容。
- 内部校准台、日志和代码注释可以保留原因；客户可见输出不保留原因。

## 产品资料增量读取规则

以后调用或分析 `docx_2` 产品资料时，不得默认重新全文读取全部文件。

1. 先读取 `docs/product-knowledge/README.md`、`confirmed-facts.md`、`conflicts.md`、`decisions.md` 和 `source-manifest.json`。
2. 运行 `npm.cmd run audit:product-docs` 比对每个源文件的 SHA-256。
3. 只处理 `work/product-doc-audit/incremental/changed-files.json` 中新增、哈希变化或仍待提取的文件；忽略 Word `~$` 锁文件。
4. 全文、图片、转换文件和增量提取缓存继续放在 Git 忽略的 `work/product-doc-audit`，Git 内只保存精简事实、冲突、用户决策、来源优先级和源文件哈希。
5. 来源冲突按 `用户确认口径 > 正式规格书与安装手册 > 产品白皮书 > 解决方案 > 营销文案` 处理，并把冲突和最终决策写回知识库。

当前 RING08 资料存在 `5m` 与 `5-8m` 表述冲突；最终业务口径按用户确认执行：线上拾音（包含互动课堂的线上互动拾音）半径 `8m`，本地扩声半径 `5m`。互动课堂同时包含本地扩声时，本地扩声能力仍按 `5m` 复核。

## 公司 Agent 询问与冲突记录规则

以后遇到产品接口、接线方式、设备能力、安装要求或其他无法从用户已确认口径和本地资料中确定的问题时，优先直接询问公司 Agent，不得自行猜测。

- 每次只问一个具体问题，不要把多个设备、多个接口或多层规则合并成抽象问题。
- 问法优先设计成 Agent 可以直接回答“是”或“不是”的形式；如果回答“不是”“不完全”或存在条件，再追问具体原因和适用边界。
- 可以先问设备的具体接口接线图，再逐项确认某个接口能否连接某类设备；不要要求 Agent 一次性制定整套售前算法或抽象制图规范。
- Agent 回答与用户此前确认口径冲突时，永远以用户确认口径为准，不得用 Agent 回答覆盖用户决定。
- 每一条冲突都必须记录到 `logs/execution_log.md`、`logs/retrospective.md` 和当前专项审计文档，写清 Agent 说法、用户口径和最终采用规则。
- Agent 明确无法回答、知识库未收录或资料不足时，不得补全、类推或借用其他产品接口；必须汇总到专项“待用户手工补充”文档，保留明确字段供用户填写。
- Agent 回答只能作为资料证据之一；正式落地前仍需与用户确认、正式手册和已确认实物图交叉核对。
